# Example: 線上程式學習平台的學員流失診斷

## Scenario

CodePath Taiwan 是一家提供後端工程師轉職訓練的線上平台。產品團隊觀察到，學員在第三週課程（「SQL JOIN 與子查詢」單元）的完課率從第二週的 78% 驟降至 41%，且大量學員在進入練習題後的 15 分鐘內就關閉頁面。

課程設計如下：

- 每段影片 8–12 分鐘，共 6 段
- 練習題：給定真實電商資料集（含 5 張資料表），撰寫 10 道 SQL 查詢
- 批改方式：提交後 24 小時內由助教回覆批改結果
- 目標：學員能獨立完成多表 JOIN + 聚合函數

產品負責人 Angela Chen 想知道：「為什麼學員在第三週大量放棄？我們應該怎麼重新設計這個單元？」

---

## Analysis

### Step 1 — 定位技能–挑戰空間

**技能評估（第三週開始時）**

學員在前兩週學完基本 SELECT、WHERE、GROUP BY，但尚未接觸多表關聯。根據平台的前測資料：

| 學員群體 | SQL 自評技能分 (1–10) | 佔比 |
|----------|----------------------|------|
| 初學者（無 DB 背景）| 3–4 | 62% |
| 有一點基礎（用過 Excel / 簡單 SELECT）| 5–6 | 31% |
| 已有工作經驗 | 7+ | 7% |

**挑戰評估（第三週練習題）**

練習題跨 5 張資料表、需要 3–4 層巢狀 JOIN，業界難度約 6–7 分（中高）。

**象限定位**：

```
高挑戰
    │
焦慮 │ ★ 第三週大多數學員（3–4 技能 vs 6–7 挑戰）
    │
────┼─────────────────
    │
冷漠 │              無聊
    │
   低技能 ──────── 高技能
```

**結論**：62% 的初學者群落在「焦慮」象限，挑戰遠超感知技能。這解釋了練習題 15 分鐘內的棄課行為——不是無聊，是習得性無助。

---

### Step 2 — 評估 Flow 前置條件

| 條件 | 狀態 | 具體證據 |
|------|------|---------|
| **清晰的近端目標** | ❌ 未滿足 | 「撰寫 SQL 查詢 10 道」沒有給出中間里程碑；學員不知道第 1 道和第 10 道之間有什麼進展 |
| **即時回饋** | ❌ 未滿足 | 助教 24 小時批改，學員提交後完全不知道自己的 JOIN 邏輯是否正確，無法當場修正錯誤認知 |
| **技能–挑戰匹配** | ❌ 未滿足 | 初學者技能分 3–4，練習題挑戰分 6–7，差距超過 2 個單位，遠超 flow channel 寬度 |

三項前置條件全部未滿足，flow 在結構上不可能發生。

---

### Step 3 — 識別 Flow 阻礙因子

1. **難度斷層（最主要）**：影片內範例只有 2 張表的 INNER JOIN，練習題直接跳至 5 張表 + LEFT JOIN + 子查詢，難度曲線出現垂直跳躍，學員無橋接點。

2. **延遲回饋迴路**：24 小時批改使錯誤無法即時被糾正，學員卡住時只能猜測或放棄，而非調整策略再試。

3. **目標模糊**：「撰寫 10 道查詢」是終點，不是里程碑。學員沒有「我現在掌握到哪一步」的感知信號。

4. **自我意識干擾**：平台論壇公開顯示每人提交次數，部分學員表示「怕被看到試了很多次還沒過」，造成評估焦慮，進一步壓縮嘗試行為。

5. **缺乏掌握度可視化**：沒有顯示「你已解鎖：2-表 JOIN」這類技能進度指標，學員無法感受到漸進成長。

---

### Step 4 — 設計建議

**核心原則**：將第三週單元切割為技能梯度分明的三個子關卡，使每個關卡的挑戰落在當前技能的 +1 區間，而非 +3。

**建議的難度曲線重設計**：

```
技能感知
 7 │                             ●── 關卡 C（5 表 + 子查詢）
 6 │                         ╱
 5 │                 ●── 關卡 B（3 表 + LEFT JOIN）
 4 │             ╱
 3 │     ●── 關卡 A（2 表 INNER JOIN）
 2 │ ╱
 1 └─────────────────────────────────────
   第一週基礎   關卡A   關卡B   關卡C
```

**具體行動項目**：

1. **挑戰分層**：將 10 道題拆為 3 關：A 關（2 題，2 表 JOIN）→ B 關（4 題，3 表 + GROUP BY）→ C 關（4 題，複雜子查詢）。每關解鎖前一關才開放。

2. **即時自動批改**：導入 SQL 執行沙箱，學員提交後 < 3 秒顯示查詢結果 vs 預期輸出的差異，附「你的 JOIN 條件正確，但 WHERE 篩選有誤，再試一次？」的引導提示，取代 24 小時人工批改。

3. **近端目標可視化**：每道題顯示「本題考點：INNER JOIN + COUNT」，完成後出現綠色打勾並顯示「已掌握：多表聚合」。

4. **移除公開提交計數**：將論壇的提交次數改為私人，降低評估焦慮；或改為顯示「班級平均嘗試次數：4.2 次」以正常化多次嘗試。

5. **掌握信號**：每完成一個關卡，生成個人化的「SQL 技能地圖」標記已解鎖節點，讓技能成長可見。

---

## Result

```markdown
## Flow Analysis: CodePath Taiwan — 第三週 SQL 課程

### Current State Diagnosis
- Perceived skill level: Low（62% 學員自評 3–4 / 10）
- Perceived challenge level: High（練習題難度 6–7 / 10）
- Current experience zone: **Anxiety（焦慮）**

### Precondition Check
| Condition | Status | Evidence |
|-----------|--------|----------|
| Clear goals | Unmet | 無中間里程碑；學員不知道階段性目標 |
| Immediate feedback | Unmet | 24 小時批改，錯誤無法即時修正 |
| Skill-challenge match | Unmet | 技能 3–4 vs 挑戰 6–7，差距 > 2 單位 |

### Flow Blockers
- 難度斷層：影片範例（2 表）→ 練習題（5 表）跳躍過大，無橋接設計
- 回饋延遲：24 小時批改使學員卡住後只能放棄
- 目標模糊：缺乏近端里程碑，學員無進度感
- 評估焦慮：公開提交次數引發自我意識干擾

### Design Recommendations
1. **挑戰重分層**：10 題拆為 A/B/C 三關（難度梯度 3→5→7），依序解鎖
2. **即時 SQL 沙箱批改**：< 3 秒回傳差異比對 + 引導提示，取代人工批改
3. **近端目標 + 掌握可視化**：每題標示考點，完成後解鎖「技能地圖」節點
4. **移除公開提交計數**：改為顯示班級匿名平均，降低評估焦慮
```

**預期影響**：若技能–挑戰差距從 3 個單位縮減至 1 個單位，並補齊即時回饋與近端目標，依 Csikszentmihalyi 的 ESM 研究，可預期 flow 發生頻率提升，完課率目標可設定為從 41% 回升至 65–70%（對標第二週基準線並保留合理難度損耗）。

> **注意**：上述設計對 C 關的高挑戰學員（7% 有工作經驗者）可能產生無聊感，建議為此群體提供選擇性「進階挑戰題」（開放式最佳化查詢），維持其 flow 平衡。
