# Pyramid Principle — Writing Guide

Barbara Minto 於麥肯錫發展出此框架，核心命題只有一個：**先給答案，再給理由。**

---

## 核心結構：金字塔

```
         ┌─────────────────┐
         │  主要建議 / 答案  │  ← 第一句話就說
         └────────┬────────┘
        ┌─────────┼─────────┐
        ▼         ▼         ▼
   [論點 1]   [論點 2]   [論點 3]    ← MECE，3-5 個
        │         │         │
      [細節]    [細節]    [細節]      ← 只在被追問時展開
```

**每一層都必須回答上一層的「為什麼？」或「如何做到？」**

---

## 三種組織邏輯

### 1. 演繹式（Deductive）
適合：觀眾已同意問題框架，你在推導結論

```
前提 1：所有高流失率都源於 onboarding 體驗差
前提 2：我們的 30 日留存率低於業界 20 個百分點
─────────────────────────────────────────────
結論：我們的 onboarding 體驗是流失主因
```

**陷阱**：前提若有爭議，結論會被連帶否定。觀眾不買單前提時，改用歸納式。

### 2. 歸納式（Inductive）
適合：多個獨立證據共同指向同一結論

```
結論（先說）：應該裁撤北區倉庫

論點 1：北區倉庫人均處理量是全國最低（2,100 件/人）
論點 2：北區租金是台中倉的 3.2 倍
論點 3：北區 95% 訂單可在 4 小時內由台中配達
```

每個論點獨立成立，合起來支撐結論。

### 3. 情境式（SCR / Situation-Complication-Resolution）
適合：說服持懷疑態度的聽眾、寫給高層的 Executive Summary

| 組成 | 說什麼 | 範例 |
|------|--------|------|
| Situation | 雙方已知的背景事實 | 「Q3 毛利率 18%，目標 25%」 |
| Complication | 打破現狀的緊張點 | 「競爭對手本季降價 15%，若不回應，Q4 客戶流失估達 30%」 |
| Resolution | 建議（答案） | 「建議啟動成本重組計畫，目標 6 個月內毛利率回到 23%」 |

SCR 不是「背景→問題→分析→結論」的敘事順序；Resolution 就是 Pyramid 的頂點，仍須先說。

## 完整寫作程序

### Step 1：定義頂點（What is your point?）
問自己一個問題，然後用一句話完整回答：

> 「如果讀者只看第一行，他應該帶走什麼？」

壞版本：「這份報告分析了用戶流失問題。」  
好版本：「應立即對 3+ 單/週的用戶免除外送費，可在 90 天內回復 15% 的流失市佔。」

### Step 2：找出支撐論點
列出所有理由。然後問：

1. **這些論點 MECE 嗎？** 有沒有重疊？有沒有遺漏的關鍵理由？
2. **每個論點夠強嗎？** 如果去掉它，結論還成立嗎？若是，刪掉。
3. **數量對嗎？** 少於 3 個通常顯得單薄；超過 5 個讀者難以記憶。

### Step 3：排序論點
三種排序原則，選其一：

| 原則 | 適用場景 | 範例 |
|------|---------|------|
| **影響力由大到小** | 說服 / 建議 | 最有力的論點第一 |
| **時間順序** | 計畫 / 流程 | 先做 A，再做 B，再做 C |
| **結構順序** | 診斷 / 分析 | 收入面、成本面、資本面 |

混用原則會讓讀者失去導航感，選定一個就貫徹。

### Step 4：寫連結句（Transition）
論點之間需要明示邏輯關係，不要讓讀者自己腦補：

```
「有三個原因支持這個建議：」          ← 歸納式連結
「因此我們可以得出：」                ← 演繹式連結
「情況已發生根本改變，因為：」         ← SCR 連結
```

### Step 5：在細節層級補充證據
只在被追問、或預知會有質疑時才在細節層展開。  
細節層的任務是**支撐論點**，不是展示工作量。

## 常見失誤對照表

| 錯誤模式 | 症狀 | 修正方式 |
|---------|------|---------|
| **屍體報告** | 開頭是背景、方法論、數據，答案埋在最後 | 把結論移到第一行 |
| **論點變描述** | 「銷售額下降了」（事實陳述，不是論點） | 改為「銷售額下降源於定價失當」（含因果） |
| **偽 MECE** | 「問題：技術、行銷、客服」（類別重疊） | 重新用 Issue Tree 拆解，確認無交集 |
| **論點過多** | 7-8 個論點平鋪 | 分組，每組不超過 5 個，再加一層 |
| **支撐變跑題** | 細節層的資料沒有直接支撐上方論點 | 問「這個數據支持哪個論點？」若答不出來，刪除 |

## 完整範例：外送 App 市佔流失

**情境**：主管問「我們的市佔為何下滑？我們該怎麼辦？」

### 壞版本（敘事順序）

> 我們分析了過去兩季的數據。首先看用戶數，新用戶獲取成長 5%，但流失率從 8% 升至 14%。接著看競爭對手，Panda 推出了免運優惠。我們的平均客單價是 NT$280，Panda 是 NT$310……（繼續 3 頁）……因此我們建議推出忠誠計畫。

問題：答案在最後，主管要讀完才知道建議是什麼。

---

### 好版本（Pyramid 順序）

**頂點（第一句）：**
> 建議針對每月訂單 12 次以上的用戶推出免運門檻計畫，預估 90 天內止血 60% 的流失、回復 NT$1.2M 月營收。

**三個論點：**

**論點 1：流失集中在高頻用戶**
- 流失率從 8% 升至 14%（+6pp）
- 其中 72% 來自原本 3+ 單/週的用戶
- 這群人貢獻我們 58% 的 GMV

**論點 2：流失與外送費調漲高度相關**
- 費用從 NT$39 調至 NT$59（+51%）的次月，此群流失率跳升 9pp
- 同期 Panda 推出「滿 NT$250 免運」方案
- 用戶調查：47% 的流失用戶提及「外送費太貴」

**論點 3：免運門檻計畫財務可行**
- 目標用戶（月訂 12 次以上）共 8.4 萬人
- 平均每單毛利 NT$42，免運成本 NT$20/單
- 若回復 60% 流失用戶，淨增 GMV 超過方案成本 3.1 倍

**下一步：**
1. 2 週內上線 A/B 測試（控制組維持現價）
2. 4 週後評估留存率變化決定全面推出

## 口頭報告的 Pyramid 應用

書面與口頭結構相同，但口頭需要額外的導航句：

```
開場（頂點）：「我今天的建議是 X，有三個理由。」
論點 1：     「第一，……（數據）……」
論點 2：     「第二，……（數據）……」
論點 3：     「第三，……（數據）……」
收尾（重申）：「所以我建議 X，最快可在 2 週內測試。」
```

不要說「我接下來要介紹背景」，直接給結論。背景只在有人問時才補充。

## 快速自我檢查清單

在送出任何分析前，問這五個問題：

- [ ] 第一句話是建議 / 答案，不是背景陳述？
- [ ] 論點之間 MECE（無重疊、無遺漏）？
- [ ] 每個論點有具體數據或事實支撐？
- [ ] 論點數量在 3-5 個之間？
- [ ] 細節層每一條都直接支撐上方論點？
