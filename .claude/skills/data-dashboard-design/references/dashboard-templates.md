# Dashboard Wireframe Templates

儀表板骨架（wireframe）決定了用戶第一眼看到什麼。以下是四種依受眾分層的實際版面配置，每種都包含格線結構、元件清單，以及具體的指標填法範例。

---

## Template 1 — Executive Dashboard（高管層）

**適用情境**：C-suite、董事會週報/月報，5 秒內回答「我們還在正軌上嗎？」

### 格線結構（12 欄 × 4 列）

```
┌──────────────────────────────────────────────────────────────────────┐
│  ROW 1 (高度: 1.5 倍)                                                 │
│  [Scorecard: North Star]  [Scorecard: Revenue]  [Scorecard: Churn]   │
│  欄 1–4                   欄 5–8                欄 9–12               │
├──────────────────────────────────────────────────────────────────────┤
│  ROW 2 (高度: 2.5 倍)                                                 │
│  [Line Chart: North Star 趨勢 12 個月]          [Bar: L1 KPI 達成率] │
│  欄 1–7                                          欄 8–12              │
├──────────────────────────────────────────────────────────────────────┤
│  ROW 3 (高度: 2 倍)                                                   │
│  [Bullet Chart: 4 個 OKR vs 目標]               [Alert 摘要表]       │
│  欄 1–7                                          欄 8–12              │
├──────────────────────────────────────────────────────────────────────┤
│  ROW 4 (高度: 1 倍)                                                   │
│  [最後更新時間]  [資料來源標籤]  [查看完整報告連結]                   │
└──────────────────────────────────────────────────────────────────────┘
```

### 元件規格

| 位置 | 元件類型 | 指標 | 比較基準 | 格式 |
|------|---------|------|---------|------|
| R1C1–4 | Scorecard | North Star（例：MAU） | MoM % change | `1.2M ▲ 8%` |
| R1C5–8 | Scorecard | MRR / Revenue | MoM + vs Budget | `$4.2M ▲ 3% / 97% of target` |
| R1C9–12 | Scorecard | Churn Rate | MoM change | `2.1% ▼ 0.3pp` (紅/綠) |
| R2C1–7 | Line Chart | North Star 月趨勢 | 同期去年對比線 | 雙線，Y軸從 0 開始 |
| R2C8–12 | Horizontal Bar | L1 KPI 達成率 | 100% target 線 | 顏色：≥90% 綠、70–90% 黃、<70% 紅 |
| R3C1–7 | Bullet Chart × 4 | OKR 進度 | 目標值 + 拉伸目標 | 每條 OKR 一列 |
| R3C8–12 | 純文字表 | 本期 Top 3 警報 | — | 紅框，含跳轉連結 |

### 填法範例（SaaS 產品）

```
North Star: MAU = 1,248,000  ▲ 8.3% MoM
Revenue:    MRR = $4,210,000 ▲ 3.1% MoM | 97% of $4.34M budget
Churn:      2.1%             ▼ 0.3pp MoM  ← 綠色（改善）

趨勢線：顯示過去 13 個月 MAU，標記 Q4 促銷活動時間點

OKR Bullet:
  Acquisition  ████████░░  82% of target
  Retention    ██████████  101% of target
  Revenue      ████████░░  97% of target
  NPS          ████░░░░░░  64% of target ← 黃色預警
```

## Template 2 — Operational Dashboard（營運層）

**適用情境**：團隊主管日常監控，回答「今天/本週哪個環節需要注意？」

### 格線結構（12 欄 × 5 列）

```
┌──────────────────────────────────────────────────────────────────────┐
│  ROW 1: 全域篩選器                                                    │
│  [日期範圍]  [產品線]  [地區]  [Team]  [Channel]                     │
├──────────────────────────────────────────────────────────────────────┤
│  ROW 2: 核心 KPI Scorecards（6 個）                                  │
│  [Orders] [GMV] [CAC] [Conv%] [AOV] [Return Rate]                   │
│  各欄 2 欄寬                                                          │
├──────────────────────────────────────────────────────────────────────┤
│  ROW 3 (高度: 3 倍)                                                   │
│  [Line: 每日訂單量 + 7天MA]    [Funnel: 漏斗轉換率]                  │
│  欄 1–6                        欄 7–12                                │
├──────────────────────────────────────────────────────────────────────┤
│  ROW 4 (高度: 3 倍)                                                   │
│  [Stacked Bar: 各渠道 GMV]     [Heatmap: 時段 × 星期 訂單密度]      │
│  欄 1–6                        欄 7–12                                │
├──────────────────────────────────────────────────────────────────────┤
│  ROW 5 (高度: 2 倍)                                                   │
│  [Table: Top 20 SKU / Campaign，可排序]                               │
│  欄 1–12                                                              │
└──────────────────────────────────────────────────────────────────────┘
```

### Funnel 元件規格

Funnel 是營運層最容易填錯的元件。每一步必須顯示：

1. **絕對數字**（本步驟人數）
2. **對上一步的轉換率**（step-to-step rate）
3. **對頂部的累積轉換率**（overall rate）

```
訪客      100,000  (100%)
  ↓  62%
加購       62,000  ( 62% overall)
  ↓  71%
結帳        44,020 ( 44% overall)
  ↓  84%
付款完成    36,977 ( 37% overall)  ← 整體轉換率
```

如果只顯示 step-to-step 而不顯示 overall，主管會誤判漏斗健康程度。

### Heatmap 填法

欄 = 星期一至日（7 欄），列 = 時段（0–23 點，24 列）。

- 顏色尺度：最低訂單量 → 白，最高 → 深藍（或品牌色）
- 顯示絕對數字而非百分比（主管需要知道量級）
- 附一列「全週合計」作為邊際匯總

## Template 3 — Diagnostic Dashboard（診斷層）

**適用情境**：分析師、工程師用於深挖問題根因，回答「為什麼這個指標掉了？」

### 格線結構（12 欄 × 6 列）

```
┌──────────────────────────────────────────────────────────────────────┐
│  ROW 1: 全域篩選器（更細粒度）                                        │
│  [日期+時間] [User Segment] [Device] [Version] [Cohort] [A/B Group] │
├──────────────────────────────────────────────────────────────────────┤
│  ROW 2: 問題定位 Scorecards                                           │
│  [Metric X 當前值]  [Metric X 7天前]  [Delta]  [% Change]  [Z-Score]│
├──────────────────────────────────────────────────────────────────────┤
│  ROW 3 (高度: 3 倍)                                                   │
│  [Line: 小時級趨勢，可縮放]    [Bar: 分 Segment 對比，同期]          │
│  欄 1–7                        欄 8–12                                │
├──────────────────────────────────────────────────────────────────────┤
│  ROW 4 (高度: 3 倍)                                                   │
│  [Scatter: 相關指標 X vs Y]    [Histogram: 分佈變化對比]             │
│  欄 1–6                        欄 7–12                                │
├──────────────────────────────────────────────────────────────────────┤
│  ROW 5 (高度: 2 倍)                                                   │
│  [Cohort Retention Grid]                                              │
│  欄 1–12                                                              │
├──────────────────────────────────────────────────────────────────────┤
│  ROW 6 (高度: 2 倍)                                                   │
│  [原始資料表：可排序、可匯出 CSV]                                     │
│  欄 1–12                                                              │
└──────────────────────────────────────────────────────────────────────┘
```

### Cohort Retention Grid 格式

```
Cohort  | Wk0  | Wk1  | Wk2  | Wk3  | Wk4  | Wk8  | Wk12
--------|------|------|------|------|------|------|------
2026-01 | 100% | 42%  | 31%  | 26%  | 23%  | 18%  | 15%
2026-02 | 100% | 45%  | 33%  | 28%  | 25%  | —    | —
2026-03 | 100% | 48%  | 35%  | —    | —    | —    | —
```

- 每格顏色深淺對應留存率（深色 = 高）
- 未完成的格子留空，**不填 0**（0 和「尚未到期」是完全不同的訊號）
- 顯示各 cohort 的初始人數（sample size），小 cohort 需標示警告

### Z-Score 異常偵測公式

診斷層 Scorecard 應顯示 Z-Score 幫助快速判斷指標下跌是否為統計異常：

```
Z = (X_today - μ_baseline) / σ_baseline

μ_baseline = 過去 28 天同星期幾的平均值（控制週期性）
σ_baseline = 過去 28 天同星期幾的標準差

規則：
  |Z| < 1.5  → 正常波動（灰色）
  1.5 ≤ |Z| < 2.5 → 需觀察（黃色）
  |Z| ≥ 2.5  → 異常，觸發告警（紅色）
```

範例計算：

```
今日訂單量：8,200
過去 28 天週三均值 (μ)：9,100
過去 28 天週三標準差 (σ)：350

Z = (8,200 - 9,100) / 350 = -2.57  → 紅色異常
```

## Template 4 — Real-Time Monitoring Dashboard（即時監控層）

**適用情境**：工程師、客服 on-call，回答「系統現在有沒有問題？」（Grafana 典型用法）

### 格線結構

```
┌──────────────────────────────────────────────────────────────────────┐
│  ROW 1: 整體健康狀態（紅/黃/綠 燈號）                                │
│  [API Status] [DB Status] [Queue Status] [CDN Status] [Payment GW]  │
├──────────────────────────────────────────────────────────────────────┤
│  ROW 2 (高度: 2.5 倍): 流量與錯誤率                                  │
│  [Line: RPS 即時]    [Line: Error Rate %]    [Big Number: P99 延遲] │
│  欄 1–4               欄 5–8                  欄 9–12                │
├──────────────────────────────────────────────────────────────────────┤
│  ROW 3 (高度: 2.5 倍): 資源使用率                                    │
│  [Gauge: CPU %]  [Gauge: Memory %]  [Line: DB Query Time]           │
│  欄 1–3          欄 4–6              欄 7–12                         │
├──────────────────────────────────────────────────────────────────────┤
│  ROW 4: 告警日誌（最新 20 筆）                                        │
│  [Time] [Severity] [Service] [Message] [Ack]                        │
└──────────────────────────────────────────────────────────────────────┘
```

### SLO 健康燈號邏輯

即時監控的燈號必須基於 SLO（Service Level Objective），而非固定閾值：

```
SLO: API P99 延遲 < 200ms，30 天滾動窗口可用率 ≥ 99.9%

燈號計算（每 1 分鐘刷新）：
  綠燈：過去 5 分鐘 Error Rate < 0.1% AND P99 < 200ms
  黃燈：過去 5 分鐘 Error Rate 0.1–1% OR P99 200–500ms
  紅燈：過去 5 分鐘 Error Rate > 1% OR P99 > 500ms
```

不要讓燈號直接觸發 PagerDuty——黃燈只通知 Slack，紅燈持續超過 5 分鐘才 page on-call。

## 跨模板共用規則

### 時間範圍標準化

所有同一頁面的圖表**必須使用相同的預設時間範圍**，否則主管會拿不同時期的數字做錯誤比較。

| 層級 | 預設時間範圍 | 快速切換選項 |
|------|------------|------------|
| Executive | 本月 MTD | 上月、本季、本年 |
| Operational | 最近 7 天 | 昨日、最近 30 天、自訂 |
| Diagnostic | 最近 24 小時 | 最近 7 天、自訂到分鐘 |
| Real-Time | 最近 1 小時 | 最近 15 分鐘、最近 6 小時 |

### Scorecard 顯示格式規範

```
格式：[數字] [方向箭頭] [變化量] [比較說明]

範例：
  $4.2M  ▲ 8.3%  vs last month
  1,248K ▼ 2.1%  vs last week
  99.7%  → +0.0pp vs SLA target

數字格式化規則：
  < 1,000       → 顯示全數字: 842
  1,000–9,999   → 加千分號: 4,210
  10K–999K      → 縮寫: 142K
  ≥ 1M          → 縮寫到小數一位: 1.2M
  百分比         → 保留一位小數: 12.3%
  貨幣           → 依業務慣例，通常 $4.2M 或 $4,210,000
```

### 顏色語義（強制一致）

同一套儀表板系統中，顏色意義必須統一，不能在某個圖用紅色表示「目標」、在另一個圖用紅色表示「警告」。

| 顏色 | 語義 | 用例 |
|------|------|------|
| 綠 | 達標 / 改善 | KPI ≥ 目標、指標朝好方向變化 |
| 紅 | 未達標 / 惡化 | KPI < 目標、系統異常 |
| 黃/橘 | 需注意 | 接近閾值、數據延遲 |
| 藍 | 中性資訊 | 趨勢線、對比基準線 |
| 灰 | 無效 / 歷史 | 未完成的 cohort 格、已停用的渠道 |

## 常見版面錯誤對照

| 錯誤做法 | 正確做法 | 原因 |
|---------|---------|------|
| 把 30 個 KPI 放在同一頁 | 依 IRON LAW 分成 3 個受眾頁面 | 一頁只服務一個受眾 |
| Scorecard 只顯示數字，無比較基準 | 一定加「vs 上期」或「vs 目標」 | 裸數字無法判斷好壞 |
| 所有圖表 Y 軸不從 0 開始 | 趨勢類圖從 0 開始；局部比較可截斷但需標示 | 截斷 Y 軸會誇大變化幅度 |
| Pie chart 有 8 個 slice | 改用 Horizontal Bar Chart | 人眼無法比較相近的角度 |
| Funnel 只顯示 step-to-step 轉換率 | 同時顯示 overall 累積轉換率 | 避免誤判漏斗中間步驟的問題嚴重性 |
| Cohort 未完成格填 0 | 留空或標示「In Progress」 | 0 是真實數據，空格是「尚未到期」 |
| 資料來源與更新時間不顯示 | 儀表板底部永遠顯示「Last updated: ...」 | 過時資料比沒有資料更危險 |
