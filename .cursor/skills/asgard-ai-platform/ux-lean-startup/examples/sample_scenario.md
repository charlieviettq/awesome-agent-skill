# Example: 台灣 B2B SaaS 採購管理工具驗證

## Scenario

**用戶問題：**

> 我們是三人團隊，都有製造業採購背景。我們發現中小製造廠的採購流程很混亂——用 LINE 詢價、Excel 追蹤訂單、月底手動對帳。我們想做一個採購管理 SaaS，但我們已經花了兩個月在設計資料庫 schema 和後台架構。投資人說先去驗證需求，但我們不確定怎麼做，也不確定該驗證什麼。

**背景資訊：**
- 已燒 NT$120,000 在開發（兼職工程師）
- 尚未接觸任何潛在客戶
- 預計再花 4 個月才能上線 MVP

---

## Analysis

### Step 1：找出最危險的假設

這個案子有很多假設，但最貴的錯誤是先建後測。列出假設並按風險排序：

| 假設 | 風險等級 | 為何危險 |
|------|----------|----------|
| 採購流程痛苦到值得付錢解決 | **極高** | 痛苦未必等於付費意願 |
| 決策者是採購主任，不是老闆 | **高** | B2B 買家認定錯誤，整個 GTM 失效 |
| SaaS 月費模式可接受 | **高** | 台灣中小製造業偏好買斷或不想訂閱 |
| 功能集合（詢價+追蹤+對帳）是對的組合 | **中** | 競品分析可驗證 |
| 可以從 LINE/Excel 匯入資料 | **低** | 技術問題，可以解決 |

**最危險假設：中小製造廠採購主任願意為減少手動對帳時間付 NT$2,000-5,000/月。**

---

### Step 2：選擇 MVP 類型

根據假設性質選最省力的測試：

- 全功能產品 → ❌ 4 個月、耗費最大、驗證太晚
- 單功能原型 → ❌ 仍需數週開發
- **Landing Page MVP** → ✅ 3 天、直接測付費意願

**Landing Page 設計決策：**
- 標題：「採購對帳從 3 天變 30 分鐘」（聚焦最大痛點）
- 說明：三個核心流程截圖（Figma mockup）
- CTA：「預購早鳥方案 NT$1,800/月（現在登記鎖定優惠）」
- 收集：公司名稱、員工數、EMAIL、電話
- **關鍵設計**：按下「立即預購」後才告知「目前開放測試名額，我們會在三天內聯絡您」——測真實意圖，不只是滑鼠點擊

---

### Step 3：Build-Measure-Learn 計畫

**Cycle 1（第 1-2 週）**

| 項目 | 內容 |
|------|------|
| Build | Landing page（Notion + Tally 表單，無需寫 code）+ 5 篇 LinkedIn 文章鎖定「台灣製造業採購管理」 |
| 觸及管道 | 1. 個人 LinkedIn 貼文 2. 台灣製造業 Facebook 社群（五金、塑膠、電子零件） 3. 直接 cold DM：從 1111/104 找採購主任職缺的公司 |
| Measure | 訪客數、CTA 點擊率、表單完成率（含公司資訊）、電話接聽率 |
| **成功門檻** | 100 位訪客中，≥8 位填完含公司名稱的表單（8% 轉換率） |
| Learn | ≥8 → Cycle 2；<8 → 訪問 10 位訪客找原因 |

**Cycle 2（第 3-4 週）**

| 項目 | 內容 |
|------|------|
| Build | 電話訪問 Cycle 1 填表者（目標 15 通），並提供「Concierge MVP」：用人工幫 3 家試點廠商做一次完整對帳 |
| Measure | 願意排時間接受訪問比率、試點廠商中完成一次完整流程的比率、訪問後願意預付的數量 |
| **成功門檻** | ≥3 家願意預付 NT$1,800 首月費用（真實付費意願測試） |
| Learn | ≥3 筆預付 → 開始建功能；<3 → Pivot 評估 |

---

### Step 4：Pivot/Persevere 判斷框架

**4 週後的決策矩陣：**

| 結果 | 判斷 | 行動 |
|------|------|------|
| ≥3 筆預付 + 訪問反饋一致 | Persevere | 開始建最小功能集（僅對帳模組） |
| 有興趣但無人預付 | 調查原因：價格？信任？流程不符？ | 可能是 Revenue Model Pivot |
| 訪問揭露痛點在「詢價」不在「對帳」 | Problem Pivot | 重新聚焦到詢價比價功能 |
| 決策者是老闆不是採購主任 | Customer Segment Pivot | 調整訊息和銷售路徑 |
| 無人填表、無人接聽 | 考慮退出或徹底重新定義問題 | 深度訪問 20 位業內人士 |

## Result

```markdown
# Lean Startup Plan: ProcureEasy — 中小製造廠採購管理

## Riskiest Assumption
中小製造廠採購主任願意為「將月底對帳從 3 天縮短至 30 分鐘」
付 NT$1,800/月的 SaaS 訂閱費。

## MVP Design
- Type: Landing Page + Concierge
- What it tests: 付費意願（非只是「有興趣」）
- Build time: 3 天（Landing Page）；2 週（Concierge 試點）
- Success metric: 4 週內取得 ≥3 筆預付款項

## Build-Measure-Learn Plan

| Cycle | Build | Measure | Learn |
|-------|-------|---------|-------|
| 1（Week 1-2） | Landing page + LinkedIn/FB 貼文 + cold DM | 訪客 100 人中表單完成率 ≥8% | 是否有真實需求存在 |
| 2（Week 3-4） | 電話訪問 15 位 + Concierge 服務 3 家試點 | ≥3 筆預付 NT$1,800 | 是否有付費意願 → 決定開發方向 |

## Pivot/Persevere Criteria
- Persevere if: 4 週內達到 3 筆預付，且訪問揭露功能需求集中
- Pivot if: 預付 <3 筆，且訪問揭露痛點在詢價比價（非對帳）
  → Solution Pivot：重新設計功能聚焦詢價模組

## 立即停止的事
- 停止開發資料庫 schema（尚未驗證任何假設）
- 先用 Notion + Tally 建 Landing Page，不寫一行 code
- 4 週後視結果決定是否重啟技術開發
```
