# Example: SaaS 訂閱平台導入電子發票系統

## Scenario

**公司**: 沐光科技股份有限公司（Muguang Tech），台北市
**產品**: B2B + B2C 混合訂閱制 SaaS（行銷自動化工具）
**現況**:
- 每月開立發票約 800 張：B2C 個人訂戶 650 張、B2B 企業客戶 150 張
- 目前仍用紙本統一發票，由會計手動開立
- 新任 CTO 要求在 2026 Q2 前全面切換為電子發票
- 工程師詢問：「我們是 SaaS 平台，訂閱續費時要怎麼自動開票？B2B 和 B2C 流程差很多嗎？載具要怎麼處理？」

---

## Analysis

### Step 1：確認業務類型與發票格式

| 客戶類型 | 佔比 | 發票格式 | 是否需要統編 |
|---------|------|---------|-----------|
| B2C 個人訂戶 | 650 張/月 | 二聯式電子發票 | 否 |
| B2B 企業訂戶 | 150 張/月 | 三聯式電子發票 | 是（必填買方統編） |

沐光屬於**混合型**，兩種格式都需要支援。訂閱制的觸發點為每次扣款成功後自動開票。

---

### Step 2：選擇整合方式

每月 800 張屬中量級、全自動化需求（扣款後立即開票），排除 Turnkey（批次上傳，不適合即時）。

| 方案 | 評估 |
|------|------|
| Turnkey | ✗ 不支援即時 API 觸發，手動上傳批次不符訂閱流程 |
| 直接接 MOF API | ✗ 需大型 IT 團隊維護憑證、金鑰輪換，3 人工程團隊負擔過重 |
| **加值服務中心 API（綠界 ECPay）** | ✓ 有完整 SDK、沙箱環境、即時開票、支援混合 B2B/B2C |

**決定：透過綠界 ECPay 加值中心 API 整合。**

---

### Step 3：B2C 載具設計

B2C 訂戶 650 張／月，需在結帳流程收集載具資訊：

```
結帳頁新增欄位：
  [ ] 不需要載具（平台保存，可中獎）
  [x] 手機條碼  →  輸入 /XXXXXXX（7位）
  [ ] 自然人憑證 → 輸入 2碼英文 + 14碼數字
  [ ] 捐贈發票   → 輸入捐贈碼（3-7碼）
```

**注意**：手機條碼需前端即時格式驗證（slash + 7 alphanumeric），並呼叫財政部條碼查驗 API 確認有效性，避免載具錯誤導致開票失敗。

---

### Step 4：B2B 開票流程

B2B 企業訂戶 150 張／月，統編為必填：

```
企業帳戶設定頁：
  公司統編（統一編號）: [________]  ← 儲存於帳戶資料
  發票抬頭:            [________]  ← 預設公司名稱
  發票寄送 Email:      [________]
```

扣款成功 → 自動帶入統編 + 公司名 → 開立三聯式電子發票 → 寄送 PDF 至指定 Email。B2B 發票**不進入中獎機制**、無需載具欄位。

---

### Step 5：字軌申請與號碼管理

沐光每月 800 張，每雙月（2 個月）需備足 1,600 張號碼：

- 向財政部申請字軌區間，建議一次申請 3,000 張（留緩衝）
- 在系統中設定**剩餘數量告警**：低於 200 張時自動發 Slack 通知
- 字軌為 2 碼英文 + 8 碼數字，由 MOF 指定，**不可自行產生**

---

### Step 6：作廢與折讓情境

訂閱 SaaS 常見退款情境：

| 情境 | 處理方式 | 時間限制 |
|------|---------|---------|
| 當期內取消訂閱（退款） | 作廢原發票（void）+ 重開或不開 | 同雙月期內 |
| 跨期退款（上期費用退還） | 開立折讓單（不得直接作廢） | 跨期後 |
| 金額輸入錯誤 | 作廢 + 重開（void-and-reissue） | 同雙月期內 |

**跨期退款是最常踩的坑**：許多工程師直接呼叫 void API，但若已過雙月截止日，void 會被平台拒絕，必須改走折讓流程。

---

### Step 7：對帳流程設計

```
每日排程（凌晨 2:00）:
  1. 從綠界 API 拉取當日已開立發票清單
  2. 比對本地資料庫已開票訂單
  3. 若有差異 → 寫入 discrepancy_log，觸發告警
  4. 輸出每日對帳報表給財務

每雙月（申報前）:
  1. 匯出雙月發票總計：銷售額、稅額
  2. 對照 401 表格欄位填報
  3. 確認作廢發票已在 MOF 平台反映
```

# E-Invoice Implementation Plan: 沐光科技股份有限公司

## Current State
- Business type: B2B + B2C Both
- Current invoicing: Paper（手動開立統一發票）
- Transaction volume: 800 張/月（B2C 650、B2B 150）

## Integration Method
- Method: API（加值服務中心）
- Provider: 綠界科技 ECPay
- Rationale: 訂閱制需即時開票，800張/月不值得維護 Direct API 所需的憑證基礎設施；綠界有完整 SDK 支援 B2B/B2C 混合開票
