# Taiwan PDPA 實作指南

個人資料保護法（個資法）施行細則、主管機關函釋及實務判例彙整。本文聚焦「如何落地執行」，不重複 SKILL.md 的法條對照表。

---

## 一、特定目的與法律依據

台灣個資法的核心邏輯與 GDPR 的「合法基礎」不同：**每一筆個資的蒐集必須對應一個法定「特定目的」**，且目的外使用原則上禁止。

### 特定目的代號（常用）

法務部公告的特定目的清單目前有 200 多項。以下是數位服務常見代號：

| 代號 | 名稱 | 典型場景 |
|------|------|---------|
| 040 | 行銷 | 發送電子報、推播通知 |
| 063 | 非公務機關依法定義務所進行個人資料之蒐集處理及利用 | 法遵申報、KYC |
| 069 | 契約、類似契約或其他法律關係事務 | 電商訂單處理 |
| 090 | 消費者、客戶管理與服務 | 客服系統、CRM |
| 091 | 消費者保護 | 退換貨、糾紛處理 |
| 111 | 就業、任用、退休管理及相關福利 | HR 系統 |
| 148 | 網路購物及其他電子商務服務 | 電商主要業務 |
| 152 | 廣告或商業行為管理 | 廣告投放、再行銷 |
| 157 | 調查、統計與研究分析 | 用戶研究、A/B 測試 |

**實務規則**：一個處理活動可以登記多個特定目的，但不能事後補填。建立資料目錄時，每行記錄都要先確認特定目的代號。

### 合法蒐集的三種路徑

```
路徑 1：當事人書面同意
  └─ 適用：敏感資料、目的外利用
  └─ 格式要求：明確告知、可辨識同意意思表示

路徑 2：法律明文允許
  └─ 適用：稅務申報、勞保投保、法院命令
  └─ 無需另取同意，但仍需告知義務

路徑 3：公務機關執行法定職務 / 非公務機關與當事人有契約關係
  └─ 適用：服務履行（訂單寄送）
  └─ 資料使用不得超出契約必要範圍
```

## 二、告知義務：何時、告知什麼

個資法第 8 條規定，**蒐集時**必須告知當事人以下事項：

1. 蒐集機關（非公務機關）的名稱
2. 蒐集之目的（對應特定目的代號）
3. 個人資料之類別
4. 個人資料利用之期間、地區、對象及方式
5. 當事人依第 3 條規定得行使之權利及方式
6. 當事人不提供個人資料時，對其權益之影響

### 告知範本（網站隱私權政策必要段落）

```markdown
【個人資料蒐集告知事項】

一、蒐集機關：{公司名稱}（統一編號：{編號}）
二、蒐集目的：
    - 消費者、客戶管理與服務（特定目的第 090 項）
    - 網路購物及其他電子商務服務（特定目的第 148 項）
    - 行銷（特定目的第 040 項）*需另取同意
三、資料類別：姓名、電子郵件、收件地址、交易紀錄、瀏覽行為
四、利用期間：自蒐集日起至契約關係終止後 {N} 年；行銷利用至同意撤回為止
五、利用地區：中華民國境內
六、利用對象及方式：
    - 物流業者（履約必要範圍）
    - 金流業者（付款處理）
    - 行銷服務商（需另取同意）
七、您得行使之權利：
    查詢、閱覽、製給複製本、補充或更正、停止蒐集處理利用、刪除
    行使方式：{聯絡信箱或表單網址}
八、不提供之影響：若拒絕提供姓名、地址，將無法完成商品配送。
```

> **注意**：行銷目的（040）必須讓當事人可以個別拒絕，不能與服務必要資料捆綁同意。

## 三、敏感個人資料的強化要求

下列類別受個資法第 6 條特別保護，**原則禁止蒐集**，例外須符合嚴格條件：

| 敏感類別 | 說明 | 可蒐集的例外 |
|---------|------|------------|
| 醫療、健康 | 病歷、診斷紀錄、基因資訊 | 醫療機構執行醫療業務；書面同意 |
| 犯罪前科 | 犯罪偵查、刑事判決 | 公務機關法定職務；書面同意 |
| 性生活 | 性傾向、性行為 | 書面同意 |
| 病歷 | 同醫療健康 | 同上 |
| 工會 | 工會會員資格 | 依法執行職務 |
| 種族 | 族裔背景 | 書面同意 |

**實務陷阱**：

- App 蒐集健康數據（步數、心率）→ 屬敏感資料，需書面同意
- 會員填寫「素食偏好」→ 可能間接揭示宗教信仰（本身亦屬敏感），需評估
- 裝置攝影鏡頭辨識面孔用於身份驗證 → 生物特徵尚無明文規範，但主管機關函釋傾向比照敏感資料處理

## 四、當事人權利的回應 SOP

個資法第 3 條賦予當事人六種權利，企業必須能夠在合理時間內回應。

### 回應時限與收費規則

| 事項 | 法定規定 | 實務建議 |
|------|---------|---------|
| 回應期限 | 第 13 條：15 日內（複雜案件可延長至 30 日） | 建立內部 SLA：簡單案件 7 日，複雜案件 15 日 |
| 收費 | 可酌收必要成本費用 | 建議第一次免費，後續複製本收工本費 |
| 代理 | 得透過法定代理人、書面委託代理 | 需驗證委託書及身份證明文件 |

### 權利請求處理流程

```
收到請求
    │
    ▼
1. 身份驗證（需與帳號資料一致，或提供政府核發證件）
    │
    ▼
2. 確認請求類型：
   查詢/閱覽 → 匯出相關資料，去識別化無關聯的他人資料
   補充/更正 → 核實後更新，保留更正前後紀錄
   停止蒐集/處理/利用 → 停止行銷利用立即生效；核心服務需說明無法停止的理由
   刪除 → 檢查是否有法定保存義務（如發票 5 年、勞工資料、KYC 資料）
    │
    ▼
3. 若涉及法定保存義務，可拒絕刪除但須告知原因與保存期限
    │
    ▼
4. 15 日內書面（含電子郵件）回覆
    │
    ▼
5. 紀錄請求日期、回覆日期、處理結果（供稽核）
```

### 合法拒絕請求的情境

企業可依第 10 條但書拒絕請求，但必須告知拒絕理由：

- 妨害國家安全或外交及軍事機密
- 妨害犯罪偵查或訴訟
- 妨害公共利益
- **妨害第三人之重大利益**（最常見：刪除請求涉及訴訟證據）
- 當事人行使權利有詐欺或不當意圖之虞

## 五、境外傳輸（跨境資料移轉）

### 個資法第 21 條規定

主管機關（國家發展委員會）在下列情形可限制境外傳輸：

1. 涉及國家重大利益
2. 國際條約或協定有特別規定
3. 接收方所在國家保護標準不足
4. 以迂迴方式傳輸規避本法

### 合規路徑決策樹

```
資料是否傳輸至境外？
    │ Yes
    ▼
接收國是否在國發會認定「保護措施完善」名單？
    │ Yes → 可傳輸，但仍需告知當事人利用地區
    │ No
    ▼
是否符合下列任一例外？
    (a) 依條約或協定
    (b) 當事人已書面同意且知悉風險
    (c) 與接收方簽訂資料保護協議（類似 GDPR SCCs）
    (d) 接收方通過認證機制
    │ Yes → 可傳輸，留存依據文件
    │ No
    ▼
向國發會申請審查（實務上極少使用）
```

**中國大陸傳輸的特殊考量**：

國發會尚未公告中國大陸為保護措施完善地區。企業若使用中國大陸服務商（如阿里雲、騰訊雲部署在中國的節點），或將資料交給中國大陸合作方，需：

1. 取得當事人書面同意，且告知中國大陸法律（如《個人信息保護法》）可能允許政府機關存取資料的風險
2. 或與接收方簽訂資料保護協議，要求對方提供等同台灣法律的保護

## 六、資料外洩通報 SOP

### 通報義務

個資法本文未明定通報期限，但個資法施行細則第 22 條及相關法規要求：

- **向主管機關通報**：「無延遲」（without undue delay），實務上以事件確認後 72 小時為基準（參照 GDPR 慣例，且金融、電信等特定行業有各自主管機關更嚴格規定）
- **通知當事人**：事件確認後應「儘速」通知受影響當事人

### 通報啟動判斷標準

```
發現疑似事件
    │
    ▼
是否涉及「個人資料」？
    │ No → 非個資法事件，依資安規定處理
    │ Yes
    ▼
是否有「洩漏、竄改、毀損、滅失或被竊取」的可能？
    │ No → 記錄評估結果，無需通報
    │ Yes
    ▼
評估風險等級：
    低：資料已加密且金鑰未洩漏 → 可能無需通知當事人，但仍需向主管機關評估是否通報
    中：未加密但資料量小、非敏感 → 需通報主管機關，審慎評估通知當事人
    高：大量個資、敏感資料、財務資料洩漏 → 立即通報主管機關 + 通知當事人
```

### 通報內容清單

```
【主管機關通報書】
□ 事件發生時間（已知或估計）
□ 事件類型（外部入侵 / 內部人員不當存取 / 系統錯誤 / 其他）
□ 受影響的個人資料類別與筆數
□ 受影響當事人數量（估計值）
□ 可能造成之損害（財務損失、名譽損害、人身安全風險）
□ 已採取之補救措施
□ 後續預計採取之措施
□ 企業聯絡窗口資訊

【當事人通知內容】
□ 事件說明（白話文，不使用技術術語）
□ 洩漏的資料類別（明確告知，不可過於籠統）
□ 可能的影響
□ 企業已採取的保護措施
□ 建議當事人採取的自我保護措施（如更換密碼、留意釣魚信件）
□ 企業諮詢窗口
```

## 七、資料保存與刪除政策

### 法定保存義務（常見）

以下資料有法律強制保存期限，**當事人要求刪除時可合法拒絕**：

| 資料類別 | 法源 | 保存期限 |
|---------|------|---------|
| 電子發票 | 電子發票實施辦法 | 5 年 |
| 勞工資料（薪資、出勤） | 勞動基準法 | 5 年 |
| 會計憑證 | 商業會計法 | 10 年 |
| KYC/AML 資料（金融機構） | 洗錢防制法 | 5 年 |
| 網路服務使用者紀錄 | 電信管理法、特定行業 | 6 個月至 1 年 |

### 刪除政策設計範本

```
【資料類別刪除規則】

類別：會員基本資料（姓名、信箱、電話）
  保存期間：帳號有效期間 + 帳號刪除後 3 年
  保存理由：消費者保護糾紛處理（69 條特定目的）
  自動刪除：帳號刪除 + 3 年後由系統自動清除

類別：交易紀錄
  保存期間：交易完成後 7 年
  保存理由：稅務申報、會計憑證法定義務
  自動刪除：7 年後由系統自動清除
  例外：涉及訴訟者保留至訴訟終結後 1 年

類別：行銷偏好 / 電子報訂閱狀態
  保存期間：同意有效期間；同意撤回立即停止利用
  刪除政策：同意撤回後 30 日內從行銷名單移除
  注意：需保留「撤回紀錄」用於舉證，不等於刪除所有資料
```

## 八、罰則對照與風險評估

### 行政罰

| 違規行為 | 罰則 |
|---------|------|
| 違反第 6 條（敏感資料） | 主管機關命限期改正，未改正：NT$20 萬～200 萬 |
| 違反告知義務（第 8 條） | 命限期改正，未改正：NT$2 萬～20 萬 |
| 違反安全維護義務（第 18 條） | 命限期改正，未改正：NT$2 萬～20 萬 |
| 違反境外傳輸限制（第 21 條） | 命限期改正，未改正：NT$2 萬～20 萬 |
| 妨礙當事人行使權利（第 11-13 條） | NT$2 萬～20 萬 |

### 刑事罰（意圖營利）

| 行為 | 刑罰 |
|------|------|
| 意圖營利非法蒐集、處理、利用個資 | 5 年以下有期徒刑 + NT$1 億以下罰金 |
| 意圖損害他人利益 | 3 年以下有期徒刑 + NT$50 萬以下罰金 |

### 民事賠償

- 第 29 條：非公務機關違反本法，對當事人造成損害，應負損害賠償責任
- **無法確定實際損害時**：法院得依情節酌定每人 NT$500 ～ 20,000 元賠償
- **集體訴訟**：20 人以上可委任訴訟代理人提起集體訴訟，實務上賠償總額可觀

## 九、隱私政策合規自檢清單

針對台灣 PDPA 的最低合規要求：

```
□ 隱私政策列明所有特定目的代號
□ 告知事項包含個資法第 8 條全部 6 項
□ 行銷同意可獨立撤回（不與服務捆綁）
□ 敏感資料另取書面同意（且同意書明確列出資料類別）
□ 境外傳輸地區已在告知事項中揭露
□ 當事人行使權利的聯絡方式清楚可見
□ 隱私政策版本與生效日期有記錄
□ 資料保存期限有書面政策
□ 資料外洩應變 SOP 存在且定期演練
□ 員工個資處理訓練有紀錄
```

## 十、常見錯誤的對應做法

**錯誤一：「用戶勾選服務條款即視為同意個資蒐集」**

不符合「自由意志」要求。服務條款同意與個資同意必須分開，且個資同意必須具體說明目的與類別。

**錯誤二：「行銷名單外購後直接使用」**

需確認原始蒐集時的特定目的是否包含「提供第三方行銷使用」，且原始告知對象須與你的業務性質相符。實務上外購名單難以確認合法性，風險極高。

**錯誤三：「刪除帳號 = 刪除所有個資」**

帳號刪除時應停止對外提供該用戶資料，但有法定保存義務的資料（交易紀錄、會計憑證）必須保留。需在隱私政策中說明此差異。

**錯誤四：「資料去識別化後即無任何限制」**

個資法第 2 條：經過處理仍能識別特定個人者為個人資料。去識別化需達到「一般人無法再識別」的程度，而非僅移除姓名欄位。使用統計揭露控制（SDC）技術並留存去識別化方法文件。

**錯誤五：「使用 Google Analytics / Meta Pixel 不需要告知」**

此類第三方追蹤工具會蒐集瀏覽行為（屬個人資料），且資料會傳輸至境外（美國）。需在告知事項中列明第三方名稱、特定目的（通常是 040 行銷或 157 研究分析），並提供當事人選擇退出的方式。
